# Responsible Disclosure Policy

**Effective Date:** June 1, 2026  
**Version:** 1.0  

## Overview

IPOReady is committed to security and protecting our users' data. We welcome responsible disclosure of security vulnerabilities and appreciate the security research community's efforts to improve our application.

## Reporting a Security Vulnerability

If you discover a security vulnerability in IPOReady, please report it to our security team immediately at **security@ipoready.ai**.

### How to Report

1. **Email:** security@ipoready.ai
   - Subject line: `[SECURITY] Vulnerability Report - <brief description>`
   - Include as much detail as possible about the vulnerability
   - Do NOT publicly disclose the vulnerability before we have a chance to fix it

2. **Contact Information:**
   - **Security Lead:** [Name]
   - **Backup Contact:** [Name]
   - **Response Time:** We aim to respond to all reports within 24 hours

## What to Include in Your Report

Please provide the following information when reporting a vulnerability:

1. **Description:** Clear description of the vulnerability
2. **Location:** Affected endpoint, component, or file path
3. **Severity:** Your assessment of the severity (Critical, High, Medium, Low)
4. **Steps to Reproduce:** Detailed steps to reproduce the issue
5. **Proof of Concept:** Code, screenshots, or video demonstrating the vulnerability
6. **Impact:** What could an attacker do with this vulnerability?
7. **Suggested Fix:** If you have a suggested remediation (optional)
8. **Your Contact Information:** Name, email, and organization

## Our Commitment

When you report a security vulnerability responsibly, we commit to:

1. **Acknowledge your report** within 24 hours
2. **Keep you informed** of our progress in fixing the vulnerability
3. **Work with you** to verify that the fix addresses the issue
4. **Provide credit** in our security advisory (if you wish)
5. **Not pursue legal action** against you if you follow this policy

## Our Responsible Disclosure Timeline

1. **Day 0:** You report the vulnerability
2. **Day 1:** We acknowledge receipt of your report
3. **Day 3:** We confirm the vulnerability and begin investigating
4. **Day 7:** We provide a timeline for when a fix will be available
5. **Day 30:** We release a patch (or explain delay)
6. **Day 31:** We publish a security advisory with credit to the researcher

**Important:** We ask that you do not publicly disclose the vulnerability until we have released a fix and provided a public advisory. This typically takes 30 days or less.

## What We Ask of You

1. **Do NOT publicly disclose** the vulnerability before we've had a chance to fix it
2. **Do NOT access data** beyond what's necessary to demonstrate the vulnerability
3. **Do NOT disrupt** our services or the experience of other users
4. **Do NOT share** the vulnerability with others without our permission
5. **Allow reasonable time** for us to develop and deploy a fix

## Safe Harbor

If you follow our Responsible Disclosure Policy, we will not:

- Take legal action against you
- Report you to law enforcement
- Hold you responsible for any damage caused by testing the vulnerability

We appreciate your responsible handling of security vulnerabilities.

## Security Advisory Process

When a vulnerability is fixed, we will:

1. Release a security update (patch release)
2. Publish a security advisory on our GitHub repository
3. Notify all affected customers via email
4. Provide clear upgrade instructions
5. Credit the researcher in the advisory (with permission)

## Bug Bounty Program

Currently, IPOReady does not have a formal bug bounty program. However, we plan to establish one in the future. Researchers who responsibly disclose security vulnerabilities may be eligible for rewards or recognition once our program is launched.

## Security Research Policy

We encourage legitimate security research. The following activities are permitted:

- **Testing on your own instance:** Set up a local instance and test thoroughly
- **Authorized penetration testing:** Contact us to arrange a formal security assessment
- **Vulnerability scanning:** Use automated tools to scan for common vulnerabilities
- **Code review:** Review publicly available source code

The following activities are NOT permitted:

- **Unauthorized access:** Do not access systems you don't have permission to test
- **Data exfiltration:** Do not download or remove data from our systems
- **Service disruption:** Do not perform DoS attacks or disrupt service
- **Social engineering:** Do not use social engineering or phishing techniques

## Vulnerability Disclosure Advisory

When we discover or are notified of a security vulnerability, we will issue a security advisory including:

- **Summary:** Description of the vulnerability
- **Affected Versions:** Which versions are vulnerable
- **Severity:** CVSS score and severity rating
- **Impact:** Who is affected and what could happen
- **Remediation:** How to fix or mitigate the issue
- **Timeline:** When the fix was released
- **Credit:** Recognition of the researcher (if applicable)
